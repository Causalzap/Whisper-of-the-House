<GuideQuickAnswer title="Quick Answer">
For logic, remember this: **AND** means all conditions must be true, **OR** means any condition can trigger, **NOT** reverses a signal, and **XOR** is for “one but not both.” Most useful beginner circuits are simple: contamination above 5% OR badtide active opens a badwater outlet; water depth above 2.0 m AND drought active releases reservoir water.

Do not wire ten conditions at once. Build one small circuit, test it, then add delay or memory only if the system needs it.

<h2 id="timberborn-automation-guide">
  Timberborn automation guide
</h2>

This **Timberborn automation guide** is for players who have working dams, reservoirs, badwater routes, factories, or housing plans and want those systems to run with less manual babysitting.

Automation is not only about placing sensors. A good circuit needs four things:

<ul className="my-6 list-disc space-y-2 pl-6">
  <li>a clear problem,</li>
  <li>the right input signal,</li>
  <li>the right logic mode,</li>
  <li>a safe output that cannot ruin the colony if it triggers.</li>
</ul>

The shortest rule is: **manual route first, circuit second**. If a badwater outlet has nowhere safe to drain, an OR relay will not save it. If a reservoir floods when opened by hand, automation will only flood it more consistently.

If you are new to Timberborn automation, do not unlock everything randomly. Start with tools that are easy to test and useful as overrides.

<h2 id="automation-components">
  Automation components: what each one is for
</h2>

Most confusion comes from using the wrong input. A Flow Sensor is not a Depth Sensor. A Weather Sensor does not tell you how much water remains. A Resource Counter does not know whether a river is contaminated.

<GuideTable
caption="Timberborn automation components and best uses."
columns={componentColumns}
rows={componentRows}
/>

The first real question is always: **what am I trying to detect?**
The second question is: **what should happen when that condition is true?**

<h2 id="logic-gates">
  Logic gates: when to use AND, OR, NOT, XOR, memory and delay
</h2>

Relays and logic parts are where Timberborn automation becomes useful. They are also where many players wire the right components in the wrong way.

<GuideTable
caption="Timberborn automation logic modes explained."
columns={logicColumns}
rows={logicRows}
/>

Use **AND** when you need safety. For example, do not release reservoir water unless the reservoir has enough depth **and** drought is active.

Use **OR** when any danger should trigger a response. For example, open a badwater outlet if contamination appears **or** badtide weather begins.

Use **NOT** when the default signal is backwards. If a building should run only when storage is low, you may need to invert a “storage is high” condition.

Use **XOR** rarely. It is for special cases where one condition is allowed, but two at the same time should not trigger the output.

Use **Memory / Flip-Flop** when you need the circuit to remember a state. This is useful for housing expansion or production modes that should stay on until a reset condition happens.

<h2 id="badwater-outlet-circuit">

Badwater is the cleanest use case for OR logic.

  <figcaption className="mt-3 text-sm text-slate-400">

Use this simple version:

<ul className="my-6 list-disc space-y-2 pl-6">
  <li>Input A: Contamination Sensor &gt; 5%.</li>
  <li>Input B: Weather Sensor = Badtide.</li>
  <li>Logic: OR Relay.</li>
  <li>Output: open the badwater outlet floodgate or throttling valve.</li>
</ul>

Why OR? Because either condition is enough. If contamination is already in the system, the outlet should open. If a badtide has started, the outlet should prepare before toxic water reaches the main river.

<h2 id="reservoir-release-circuit">
  Mini circuit 2: drought reservoir release
</h2>

Reservoir release is usually an AND problem. You do not want to release water just because drought is active. You also do not want to release water just because the reservoir is full during normal weather.

A safer pattern is:

<ul className="my-6 list-disc space-y-2 pl-6">
  <li>Input A: Depth Sensor &gt; 2.0 m.</li>
  <li>Input B: Weather Sensor = Drought.</li>
  <li>Logic: AND Relay.</li>
  <li>Stability: short timer or delay.</li>
  <li>Output: open throttling valve or adjust floodgate.</li>
</ul>

This circuit says: release water only when there is enough stored depth and the season actually needs it.

<figure className="my-8">

  <figcaption className="mt-3 text-sm text-slate-400">
    Use a depth or stream reading where the reservoir problem actually happens. A sensor on the wrong tile gives the right signal for the wrong place.
  </figcaption>
</figure>

If the valve flickers, the threshold is probably too close to the water level that the valve itself changes. Add delay or use a wider target range.

<h2 id="delay-and-flicker">
  Timers and delays: stop flickering circuits
</h2>

Flickering happens when the output changes the same value that the sensor reads.

Example: a valve opens because water depth is high. The valve lowers the water depth. The sensor turns off. The valve closes. The depth rises again. The loop repeats.

<figure className="my-8">
  <Image
    src="/images/timberborn/timberborn-automation-delay-timer.webp"
    alt="Timer and delay setup used to prevent automation flicker in Timberborn"
    width={1600}
    height={900}
    className="rounded-xl"
  />
  <figcaption className="mt-3 text-sm text-slate-400">
    Add a timer or delay when a water-level circuit switches too quickly around the same threshold.
  </figcaption>
</figure>

Use a timer or delay when:

<ul className="my-6 list-disc space-y-2 pl-6">
  <li>a valve opens and closes every few seconds,</li>
  <li>water depth hovers near the threshold,</li>
  <li>a gate changes the level that controls the gate,</li>
  <li>a building turns on and off because storage is exactly near the limit.</li>
</ul>

Do not add ten more relays to fix flicker. First add time or widen the threshold.

<h2 id="powered-building-circuit">
  Mini circuit 3: powered building thresholds
</h2>

Powered buildings can waste power when outputs are full or inputs are missing. A threshold circuit pauses production before it burns power for nothing.

<figure className="my-8">
  <Image
    src="/images/timberborn/timberborn-automation-powered-building-thresholds.webp"
    alt="Resource counter and threshold automation for powered buildings in Timberborn"
    width={1600}
    height={900}
    className="rounded-xl"
  />
  <figcaption className="mt-3 text-sm text-slate-400">
    Resource counters let you pause powered buildings when the output is already full or the input chain is empty.
  </figcaption>
</figure>

Use this pattern:

<ul className="my-6 list-disc space-y-2 pl-6">
  <li>Input A: output storage above your target reserve.</li>
  <li>Input B: input resource below your safe reserve.</li>
  <li>Logic: OR Relay.</li>
  <li>Output: pause the powered building.</li>
</ul>

Why OR? Either condition is enough to pause. If output is already high, production is wasteful. If input is too low, production cannot run properly.

Add memory if you want the building to stay paused until the output drops below a lower reset threshold. That avoids rapid on/off switching near the same storage number.

Population automation is useful, but it should not be your first automation project. It matters after food, water, and jobs are stable.

A safe version needs multiple conditions:

<ul className="my-6 list-disc space-y-2 pl-6">
  <li>Input A: free beds = 0.</li>

  <li>Memory / Flip-Flop: keep the next housing block enabled after it opens.</li>
  <li>Output: enable the next housing section.</li>
</ul>

This is an AND circuit because all conditions should be true. Free beds alone should not trigger more housing if the colony is already short on water.

<h2 id="automated-floodgates">
  Automated floodgates: keep it simple first
</h2>

Automated floodgates are powerful for drought and badtide control, but they are also easy to overbuild.

<figure className="my-8">
  <Image
    src="/images/timberborn/timberborn-automated-floodgate-dam.webp"
    alt="Automated floodgate set to close during non-temperate weather in Timberborn"
    width={1600}
    height={900}
    className="rounded-xl"
  />
  <figcaption className="mt-3 text-sm text-slate-400">
    Weather-based floodgate control is useful only after the dam or reservoir already behaves correctly by hand.
  </figcaption>
</figure>

Start with one clean behavior:

<ul className="my-6 list-disc space-y-2 pl-6">
  <li>close during non-temperate weather,</li>
  <li>open only when upstream depth is high enough,</li>
  <li>close if downstream water becomes unsafe,</li>
  <li>release slowly through a valve if gate drops waste too much water.</li>
</ul>

If the gate is trying to store water, release water, prevent flooding, and block badwater all at once, split the job into separate circuits.

<h2 id="faq">
  Timberborn automation FAQ
</h2>

### What automation component should I unlock first?

Start with a Lever, then a Depth Sensor or Stream Gauge, then Relay. This gives you manual control, measurement, and simple logic before complex circuits.

### What is the difference between AND and OR in Timberborn automation?

AND triggers only when every condition is true. OR triggers when any condition is true. Use AND for safety checks and OR for danger responses.

### What is a good first badwater automation circuit?

Use Contamination Sensor > 5% OR Badtide Weather as inputs, then open the badwater outlet valve or floodgate.

### When should I use a timer or delay?

Use a timer or delay when a valve, floodgate, or building flickers because the output changes the same value the sensor reads.

### Is a Flow Sensor the same as a Depth Sensor?

No. Flow Sensor checks whether water is moving. Depth Sensor or Stream Gauge checks water level. Use the one that matches the problem.

### Should I automate population growth?

Only later. Use population automation after food, water, jobs, and housing support are stable. It is not a first survival tool.

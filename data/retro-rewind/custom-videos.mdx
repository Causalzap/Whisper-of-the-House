<h2 id="how-to-add-custom-videos-quick-steps">
  How to add custom videos (quick steps)
</h2>

1. Find the file: `RR_Channel_Public.mp4`  
2. Go to: `SteamLibrary\steamapps\common\RetroRewind\Content\Movies\VHS\Public`  
3. Replace the file with your own MP4 video  
4. Fully close and relaunch the game  
5. Launch the game again and play it on the TV

<h2 id="best-video-format-and-settings">
  Best video format and settings
</h2>

The safest setup is still:

- **MP4 container**  
- **H264 video encoding**  
- **AAC audio**  
- **512×512 resolution**

That said, **512×512 is best treated as the safest recommendation, not a hard rule**.

In practice, **encoding matters more than resolution**. Community testing backs this up — some Steam players report that videos still work without following strict resolution rules, as long as the format is correct.

The game appears to care more about **H264 + AAC + correct filename** than one exact resolution. Some players report that higher-resolution videos (including 1080p) can still work when encoded correctly.

That means 512×512 should be treated as the most reliable starting point — not the only valid setup.

If your video does not work, the issue is usually encoding — not the resolution.

Good practical habits:
- test a short clip first  
- keep the filename exact  
- avoid unusual codecs  
- use a clean MP4 export before trying anything more complicated  
- try **30 FPS** if your current export keeps failing  

<h2 id="is-there-a-file-size-limit-for-custom-videos">
  Is there a file size limit for custom videos?
</h2>

There does **not** seem to be a confirmed hard **200MB** limit.

That number gets repeated a lot, but community reports point in a different direction. Players have reported a **376MB file with about 1 hour of old movie trailers** working correctly, and other community discussion points to much larger files being usable when the encoding is right.

That strongly suggests the real issue is usually **encoding, compatibility, or video setup** rather than a strict tiny file cap.

That does **not** mean bigger is always better.

Large files can still create practical problems:
- longer export times  
- slower testing  
- more chances for encoding mistakes  
- larger storage use for no real gameplay benefit  

<GuideCallout title="Practical advice">
Do not optimize for “smallest possible file.”  
Optimize for **correct filename, H264 video, AAC audio, and a clean MP4 export**.
</GuideCallout>
